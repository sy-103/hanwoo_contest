# 1부. 하이퍼파라미터 튜닝

## 1. 하이퍼파라미터란 다시

[00-A](00A_데이터분석이란.md)에서 배웠죠. **파라미터**는 모델이 학습으로 스스로 찾는 값(계수·트리 분기), **하이퍼파라미터**는 학습 *전에* 사람이 정하는 설정입니다. 부스팅의 하이퍼파라미터 예:

| 하이퍼파라미터 | 무엇을 정하나 | 크게 하면 |
| --- | --- | --- |
| `learning_rate` | 한 걸음 보폭 | 빨리 배우지만 거칠다 |
| `n_estimators` | 트리 개수 | 더 많이 배우지만 느리고 과적합 위험 |
| `max_depth`/`num_leaves` | 트리 복잡도 | 복잡한 패턴 잡지만 과적합 위험 |
| `min_child_samples` | 잎 하나의 최소 표본 | 더 일반적(과적합 ↓) |
| `subsample`/`colsample` | 트리마다 쓰는 행·열 비율 | (작게) 무작위성↑ → 과적합 ↓ |
| `reg_alpha`/`reg_lambda` | 규제 강도(계수 억제) | (크게) 단순해짐 → 과적합 ↓ |

**왜 튜닝이 필요한가**: 이 설정에 따라 성능이 크게 바뀝니다. 기본값이 우리 데이터에 최적일 리 없으니, **더 좋은 조합을 찾는** 게 튜닝입니다.

## 2. 과적합 ↔ 과소적합의 줄다리기

튜닝의 본질은 **과적합과 과소적합 사이의 균형**을 찾는 것입니다.

```
과소적합(underfit)        딱 좋음(good fit)        과적합(overfit)
모델이 너무 단순    ←─────────────────────→    모델이 너무 복잡
패턴을 못 잡음                                  데이터를 외워버림
학습·검증 다 낮음                               학습 높고 검증 낮음
```

- **트리를 깊게/많이 = 복잡 = 과적합 쪽.** 학습 점수는 오르는데 검증 점수가 떨어지기 시작하면 과적합 신호.
- **트리를 얕게/적게 = 단순 = 과소적합 쪽.** 둘 다 낮음.

위 표에서 "과적합 ↓"라고 적힌 손잡이들(min_child_samples↑, subsample↓, 규제↑)이 **복잡도를 누르는** 장치입니다. 튜닝은 이걸 적절히 섞어 **검증 점수가 최고인 지점**을 찾는 일입니다.

## 3. 조기 종료(Early Stopping) — 자동 과적합 방지

부스팅에서 특히 유용한 장치. 트리를 1개씩 쌓을 때마다 **검증 점수를 지켜보다가, 더 이상 안 좋아지면 멈춥니다.**

```
트리 개수:  100   200   300   400   450   500 ...
검증 점수:  0.40  0.44  0.46  0.46  0.46  0.45 ← 더 쌓으니 떨어지기 시작
                            ↑ 여기서 멈춤 (조기 종료)
```

`n_estimators`를 넉넉히(예: 500) 주되, "50개 연속 개선 없으면 멈춰"라고 하면 모델이 **최적 트리 수를 스스로** 찾습니다. 과적합 직전에 멈추는 똑똑한 장치라, 우리 코드도 다 씁니다.

## 4. 좋은 조합을 찾는 방법 3가지

손잡이가 10개쯤 되고 각각 범위가 넓으니, 조합이 수백만 가지입니다. 어떻게 뒤지나?

### ① 그리드 서치(Grid Search) — 격자로 전부

각 손잡이의 후보값을 정해놓고 **모든 조합**을 다 시도. learning_rate 3개 × depth 3개 × … = 격자의 모든 점.
- **장점**: 빠짐없음. **단점**: 조합이 **폭발**합니다(3개씩 10손잡이 = 6만 가지). 비현실적.

### ② 랜덤 서치(Random Search) — 무작위로 듬성듬성

조합을 **무작위로** 골라 정해진 횟수만 시도.
- 의외로 그리드보다 효율적입니다. 중요한 손잡이의 다양한 값을 더 잘 훑기 때문(그리드는 안 중요한 손잡이에 시간 낭비).

### ③ 베이지안 최적화(Bayesian Optimization) — 똑똑하게 (우리가 쓰는 Optuna)

**"지금까지 결과를 보고, 다음에 시도할 유망한 조합을 추측"** 합니다. 무작정 찍지 않고 **학습**합니다.

비유: 보물찾기. 랜덤 서치는 아무 데나 파지만, 베이지안은 **"여기 근처에서 금이 나왔으니 주변을 더 파보자"** 라고 똑똑하게 좁힙니다. 적은 횟수로 좋은 답에 도달합니다.

**Optuna**가 이 방식의 도구입니다. 우리는 Optuna로 LightGBM 손잡이를 자동 탐색합니다.

```
Optuna가 하는 일:
1. 손잡이 조합 하나 제안
2. 그걸로 모델 학습 → 검증 Macro-F1 측정
3. 결과를 기억하고, 더 나아 보이는 다음 조합 제안
4. 정해진 횟수(N_TRIALS) 반복 → 최고 조합 반환
```

> **튜닝의 철칙**: 튜닝은 **검증셋(교차검증) 안에서만** 합니다. **holdout이나 test로 튜닝하면 안 됩니다**([C2](C2_데이터분할과_누수.md)). holdout으로 손잡이를 고르는 순간, holdout이 두 번째 학습셋이 되어 "처음 보는 데이터 점수"의 의미를 잃습니다.

## 5. 무엇부터 튜닝하나 (우선순위)

손잡이가 많아도 중요도가 다릅니다. 효과 큰 순:
1. `learning_rate` ↓ + `n_estimators` ↑ (조기 종료와 함께) — 거의 항상 약간 개선.
2. `num_leaves`/`max_depth` — 데이터가 크면 복잡도 여유.
3. `min_child_samples` — 소수 등급 과적합 조절.
4. 규제(`reg_alpha`/`reg_lambda`), `subsample`/`colsample`.

# 2부. 모델 해석 — SHAP

## 6. 왜 "해석"이 필요한가

부스팅은 트리 수백 개라 **속을 들여다볼 수 없는 블랙박스**입니다. "이 소를 1++A로 예측했다"는 알지만 **왜** 그랬는지는 안 보입니다. 그런데 우리에겐 "왜"가 중요합니다.

- **신뢰**: 모델이 합리적 이유로 판단하는지(누수나 버그가 아닌지) 확인.
- **인사이트**: "어떤 요인이 등급을 좌우하나"는 그 자체가 보고서 결과(농가 관리 포인트).
- **디버깅**: 엉뚱한 변수가 1등이면 뭔가 잘못된 신호.

## 7. 변수 중요도 — 여러 방법

"어느 변수가 중요한가"를 재는 방법은 여러 가지입니다. 넓게 봅시다.

| 방법 | 아이디어 | 한계 |
| --- | --- | --- |
| 트리 빈도/이득 | 그 변수로 몇 번, 얼마나 깊게 쪼갰나 | 방향(↑↓) 모름, 모델마다 기준 다름 |
| 순열 중요도(permutation) | 그 변수를 **뒤섞었을 때** 성능이 얼마나 떨어지나 | 직관적이나 느림 |
| **SHAP** | **각 변수가 예측을 얼마나·어느 방향으로 밀었나** | 계산 무겁지만 가장 정밀·일관적 |

SHAP이 가장 강력해서 우리가 씁니다.

## 8. SHAP의 핵심 아이디어 — 공정한 공로 배분

**SHAP**(SHapley Additive exPlanations)은 게임이론의 **섀플리 값(Shapley value)** 에서 왔습니다. 원래 질문은 이거였습니다.

> "여러 명이 협력해 성과를 냈을 때, **각자의 기여를 어떻게 공정하게 나누나?**"

이걸 예측에 옮기면:

> "여러 변수가 협력해 '1++A 확률 0.8'을 만들었을 때, **각 변수가 그 0.8에 얼마나 기여했나?**"

### 직관: 기준선에서 출발해 한 변수씩 더해보기

한 소의 예측을 이렇게 분해합니다.

```
전체 평균 예측 (기준선)          : 0.30
  + 몸무게가 450이라서           : +0.15
  + daily_gain이 높아서          : +0.20
  + 더위 노출이 보통이라서        : +0.02
  + 성별이 거세라서              : +0.08
  + ... (나머지 변수들)          : +0.05
  ─────────────────────────────────────
  = 이 소의 최종 예측            : 0.80
```

각 변수 옆의 **+0.15 같은 값이 그 변수의 SHAP 값**입니다. "기준선 0.30에서 출발해, 이 소의 변수들이 하나씩 예측을 밀어서 0.80까지 갔다"는 이야기죠. **양수면 그 클래스 확률을 올리고, 음수면 내립니다.** 그리고 모든 SHAP 값을 더하면 정확히 예측값이 됩니다(그래서 "Additive").

> 섀플리 값은 "어떤 순서로 변수를 더해도 공정하게" 기여를 나누도록 수학적으로 보장됩니다. 그래서 트리 빈도 같은 방법보다 일관적이고 믿을 만합니다.

## 9. SHAP으로 보는 두 가지 그림

### ① 전체 변수 중요도 (막대그래프)

모든 소의 SHAP 절댓값을 변수별로 평균 → "전반적으로 어느 변수가 예측을 많이 흔드나". 막대가 길수록 중요.

> **읽는 법**: 이 순위를 EDA 결과와 대조하세요. 13~17회차에서 daily_gain(등급 상관 +0.522)·WEIGHT(+0.475)·성별이 강했는데, SHAP 상위가 이와 비슷하면 모델이 합리적으로 배운 것. 뜬금없는 변수가 1등이면 누수·버그 의심.

### ② 요약 그림(summary plot) — 방향까지

한 클래스(예: 1++A)에 대해, 점 하나가 소 하나. **가로 위치 = SHAP 값(좌:확률 내림, 우:올림)**, **점 색 = 변수값 높낮이(빨강:높음, 파랑:낮음)**.

이걸로 이런 문장을 만듭니다: *"daily_gain이 **높을수록(빨강)** 1++A 확률을 **올린다(오른쪽)**"* — 변수가 어느 방향으로 작용하는지가 보입니다. 단순 중요도(크기)를 넘어 **방향과 패턴**까지 읽는 거죠.

## 11. 핵심 정리

- **하이퍼파라미터** = 사람이 정하는 설정. 튜닝 = 검증 점수가 최고인 조합 찾기 = **과적합↔과소적합 균형**.
- **조기 종료**: 검증 점수가 안 좋아지면 트리 쌓기를 자동 중단 → 과적합 방지.
- 탐색법: 그리드(전부·폭발), 랜덤(무작위·의외로 효율), **베이지안/Optuna(이전 결과로 똑똑하게 좁힘)**.
- 튜닝은 **반드시 교차검증 안에서.** holdout/test로 튜닝 금지.
- **SHAP** = 각 변수가 예측을 **얼마나·어느 방향으로** 밀었는지 공정하게 분해(섀플리 값). 합치면 예측값.
- SHAP 중요도는 **EDA와 대조**해 모델의 합리성을 점검. 요약 그림으로 **방향**까지 읽는다.
