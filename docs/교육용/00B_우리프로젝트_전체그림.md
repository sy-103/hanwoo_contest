# 00-B. 우리 프로젝트의 전체 그림

> **이 강의의 목표**: 우리가 다루는 한우 데이터가 어떻게 생겼는지, "트랙1·트랙2"가 무슨 말인지, 데이터가 왜 세 종류(train·holdout·test)인지, 그리고 분석에서 가장 위험한 함정인 "정답 누수"가 무엇인지 큰 그림으로 잡습니다. 앞 강의 [00-A](00A_데이터분석이란.md)의 피처·타깃 개념을 씁니다.


> 🗺️ **학습 여정**: 기초(00A·00B) → 데이터준비(D1·D2) → EDA(E1·E2·E3) → 〔분류 C1–C7〕 · 〔회귀 R1–R8〕  ·  **📍 지금: 기초 2/2**

---

## 1. 우리가 푸는 문제

**대회 주제**: 기상(날씨)이 한우에 어떤 영향을 주는가, 그리고 한우 등급을 예측할 수 있는가.

우리 데이터는 **한우 2,408,699마리**의 기록입니다. 한 마리당: 어디서 키웠고(지역), 언제 태어나 언제 도축됐고(날짜), 몸무게·월령·성별이 얼마고, 사는 동안 더위(THI)에 며칠 노출됐고, 최종 등급은 무엇이고 얼마에 팔렸는지… 이런 정보가 92개 열에 담겨 있습니다(`step11_features.csv`).

이걸로 **두 가지 다른 질문**에 답합니다. 이게 "두 트랙"입니다.

---

## 2. 트랙1과 트랙2 — 목적이 다른 두 분석

| | **트랙1 (분류팀)** | **트랙2 (회귀팀)** |
| --- | --- | --- |
| 질문 | "이 소의 **등급**은 무엇인가?" | "**더위**는 한우 **가격**을 얼마나 움직이나?" |
| 타깃 | 등급 (LAST_GRADE) | 가격 (COST_AMT) |
| 방법 | 분류 | 회귀 |
| 진짜 목적 | **잘 맞히기** (대회 제출 점수) | **잘 설명하기** (효과의 크기·방향) |
| 산출물 | 등급 예측값 | "더위 1%p↑ → 가격 N% 변화" 같은 문장 |

이 차이가 생각보다 큽니다. **트랙1은 "맞히면 장땡"** 이라 무슨 변수를 쓰든 점수만 높으면 됩니다. 반면 **트랙2는 "왜, 얼마나"를 설명**해야 하므로, 어떤 변수를 넣느냐가 결과의 *의미*를 바꿉니다. (이 차이 때문에 같은 변수라도 트랙1에선 쓰고 트랙2에선 빼는 경우가 생깁니다 — R1에서 자세히.)

---

## 3. 데이터는 세 종류다 (가장 많이 헷갈리는 부분)

처음 배우는 사람이 거의 항상 헷갈리는 지점입니다. 천천히 봅시다.

[00-A](00A_데이터분석이란.md)에서 "모델은 학습 데이터가 아니라 **새 데이터**를 잘 맞혀야 한다(일반화)"고 했습니다. 그러려면 **새 데이터인 척하는 시험지**가 필요합니다. 그래서 데이터를 나눕니다.

```
┌─────────────────────────────────────────────────────────┐
│  train (정답 있음, 240만)  ← 우리가 가진 학습용 전체     │
│  ┌──────────────────────────┬──────────────────────┐    │
│  │  dev (개발용, 약 80%)     │  holdout (약 20%)     │    │
│  │  → 학습·튜닝에 사용        │  → 마지막에 딱 한 번  │    │
│  │                          │     채점 (자체 모의고사)│   │
│  └──────────────────────────┴──────────────────────┘    │
└─────────────────────────────────────────────────────────┘

        ┌─────────────────────────────────────────┐
        │  test (정답 없음, 45만) ← 대회 제출용      │
        │  우리가 예측해서 채워 제출 → 주최측이 채점  │
        └─────────────────────────────────────────┘
```

| | 무엇 | 정답 | 역할 |
| --- | --- | --- | --- |
| **dev** | train의 80% | 있음 | 모델을 **학습**시키고 설정을 **튜닝**한다 |
| **holdout** | train의 20% | 있음 | 다 끝난 뒤 **딱 한 번** 채점 — "처음 보는 데이터 흉내" |
| **test** | 별도 파일 | **없음** | **대회 제출** — 우리가 예측값을 채워 넣을 진짜 새 데이터 |

### 왜 holdout을 따로 떼나?

만약 학습에 쓴 dev로 채점하면, 그건 **기출문제로 시험 보는 것**과 같아서 점수가 뻥튀기됩니다(외운 걸 그대로 맞히니까). 그래서 학습에 **한 번도 안 쓴** holdout으로 채점해야 "진짜 실력"이 나옵니다.

### 그럼 test는?

test는 정답이 아예 없습니다(주최측만 압니다). 그래서 우리가 채점을 못 합니다. **holdout 점수가 곧 test에서 받을 점수의 예상치** 역할을 합니다. holdout은 "내가 만든 모의고사", test는 "진짜 수능"인 셈입니다.

> **규율 하나**: holdout은 **딱 한 번만** 봅니다. holdout 점수를 보고 모델을 또 고치면, holdout이 사실상 "연습 시험"이 되어버려서 "처음 보는 데이터 점수"라는 의미를 잃습니다. (이 원리는 C2에서 깊이 다룹니다.)

---

## 4. 가장 위험한 함정: 정답 누수 (Data Leakage)

데이터 분석에서 **초보자가 가장 자주, 가장 크게 망하는** 실수입니다. 이름을 꼭 외우세요: **누수(leakage)**.

### 비유: 답이 인쇄된 시험지

시험지 뒤에 정답이 실수로 인쇄돼 있다고 합시다. 그걸 보고 100점을 맞으면? 실력이 아니라 **컨닝**입니다. 진짜 시험(정답 없는 test)에선 점수가 폭락하죠.

데이터에서 누수란 **"실제 예측 시점에는 알 수 없는 정보를, 모델 학습에 몰래 써버리는 것"** 입니다.

### 우리 데이터의 진짜 사례

한우 등급은 소를 **도축한 뒤** 등심 단면을 보고 매깁니다. 그런데 우리 데이터엔 **근내지방도·등심단면적·조직감** 같은 "육질 측정값"도 있는데, **이것들도 도축 후에 칼로 잘라서 잰 값**입니다. 즉 등급과 거의 같은 시점·같은 부위에서 나온 **"정답의 일부"** 입니다.

이걸 모델에 넣으면? 등급과의 상관이 0.9를 넘습니다(거의 정답). 점수는 치솟지만, **진짜 test에는 이 값이 없습니다.** 컨닝하다가 실전에서 무너지는 겁니다.

### 누수를 판단하는 단 하나의 기준

> **"도축 전이냐 후냐"가 아니라, "진짜 test 데이터에 그 값이 있느냐"** 가 기준입니다.

- 육질 측정값·WGRADE·가격 → test에 **없음** → **쓰면 누수** (분류팀 금지)
- 몸무게(WEIGHT)·월령(AGE) → 도축 후 값이지만 test에 **있음** → **써도 됨** (트랙1 합법!)

test 13개 컬럼은 이렇습니다: `지역3개, 관측소, 도축일, 판정일, 성별, 몸무게, 월령, 출생일, 소번호, 농장번호, (등급=빈칸)`. **이 13개로 계산할 수 있는 것만** 분류 모델에 쓸 수 있습니다.

> 이 누수 개념은 우리 프로젝트 전체를 관통하는 가장 중요한 원칙입니다. C2에서 "전처리 누수", "시간 누수" 등 여러 종류로 다시 만납니다.

---

## 5. 분류팀이 마주칠 또 하나의 벽: 불균형

우리 등급 16개는 **개수가 엄청나게 불균형**합니다.

- 가장 많은 1++B: **319,588마리** (13.3%)
- 가장 적은 등외: **5,540마리** (0.23%)
- 차이가 **58배**

이게 왜 문제냐면 — 모델이 게으름을 피워서 **"다 흔한 등급이라고 찍어도"** 정확도가 그럴듯하게 나옵니다. 소수 등급을 통째로 포기하는 거죠. 그래서 분류팀은 **정확도를 믿으면 안 되고**(C1), **소수 등급에 가중치를 주는 장치**(C2)가 필요합니다. 이 "불균형 다루기"가 분류팀의 큰 숙제입니다.

---

## 6. 회귀팀이 마주칠 벽: 상관 ≠ 인과

회귀팀의 핵심 난관을 미리 맛봅니다. EDA(15회차)에서 이상한 결과가 나왔습니다.

> **"더위를 많이 겪은 소가 오히려 등급이 좋다"**

상식과 반대죠(더위 = 스트레스 = 나쁨일 텐데). 여기서 **"더위가 좋다"고 결론 내리면 큰 실수**입니다. 진짜 원인은 따로 있습니다 — 더운 지역에 *마침* 사육 기술 좋은 우량 농가가 많았던 겁니다(지역이라는 숨은 변수). 이런 숨은 변수를 **교란변수**라 하고, "상관이 있다고 원인인 건 아니다(상관 ≠ 인과)"가 회귀 분석의 핵심 교훈입니다. 회귀팀은 이 교란을 **통제**하는 법을 R1에서 배웁니다.

---

## 7. 두 팀이 공유하는 것

분류팀과 회귀팀은 따로 일하지만 **두 가지를 똑같이 씁니다**.

1. **같은 데이터** (`step11_features.csv`)
2. **같은 분할** (`split_assignment.csv` — 누가 dev이고 누가 holdout인지 적힌 파일)

왜 분할을 공유하냐면, 두 팀이 다른 기준으로 데이터를 나누면 결과를 비교할 수 없기 때문입니다. **같은 시험 범위로 공부해야 점수를 비교할 수 있는 것**과 같습니다. 그래서 이 분할 파일은 **절대 새로 만들지 않습니다.**

---

## 8. 핵심 정리

- 우리 데이터는 한우 240만 마리 × 92변수. 두 질문에 답한다.
- **트랙1(분류)** = 등급 맞히기(제출), **트랙2(회귀)** = 더위→가격 효과 설명.
- 데이터는 셋: **dev**(학습·튜닝), **holdout**(자체 채점, 딱 한 번), **test**(정답 없는 제출용).
- **누수** = 컨닝. 기준은 "**test에 그 값이 있느냐**". 육질·가격은 금지, 몸무게·월령은 허용.
- 분류팀의 벽 = **불균형 58배**(정확도 함정), 회귀팀의 벽 = **상관≠인과**(교란).
- 두 팀은 **같은 데이터·같은 분할**을 공유한다. 분할은 새로 만들지 않는다.
