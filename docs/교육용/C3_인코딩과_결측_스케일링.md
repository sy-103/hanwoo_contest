# C3. 인코딩·결측치·스케일링 — 데이터를 모델이 먹을 수 있게

> **이 강의의 목표**: 모델은 숫자만 먹습니다. 그런데 우리 데이터엔 "전라남도" 같은 글자, 비어 있는 칸(결측), 단위가 제각각인 숫자가 섞여 있습니다. 이걸 모델이 먹을 수 있게 바꾸는 세 가지 작업 — **인코딩(글자→숫자)·결측 처리·스케일링** — 을 원론부터 배웁니다. 특히 인코딩은 **방법이 여러 개**라, 각각이 무엇이고 언제 쓰는지를 넓게 다룹니다.
> **앞 강의**: [C2](C2_데이터분할과_누수.md)의 누수(특히 전처리 누수·타깃 누수).


> 🗺️ **학습 여정**: 기초(00A·00B) → 데이터준비(D1·D2) → EDA(E1·E2·E3) → 〔분류 C1–C7〕 · 〔회귀 R1–R8〕  ·  **📍 지금: 분류 3/7**

---

## 1. 왜 이 작업이 필요한가

모델(특히 수학으로 돌아가는 모델)은 결국 **숫자 계산**입니다. `0.3 × 몸무게 + 0.5 × 월령 + …` 같은 식이죠. 그런데 "전라남도"를 어떻게 곱하나요? 빈 칸은 어떻게 더하나요? 그래서 학습 전에 **모든 걸 적절한 숫자로** 만들어야 합니다. 이 준비 작업을 통틀어 **전처리(preprocessing)** 라 하고, 여기선 그중 셋을 봅니다.

```
글자(범주형)  → 인코딩으로 숫자화
빈 칸(결측)   → 채우거나, 그대로 두거나 (모델에 따라)
들쭉날쭉 단위 → 스케일링으로 크기 맞춤 (일부 모델만 필요)
```

---

## 2. 변수의 두 종류: 수치형 vs 범주형

먼저 변수를 구분합시다. 인코딩은 **범주형**에만 합니다.

- **수치형(numerical)**: 크기·순서에 의미가 있는 숫자. 몸무게(450kg), 월령(31개월), 더위 노출 비율(0.42). **그대로 쓰면 됩니다.**
- **범주형(categorical)**: 종류·이름을 나타내는 값. 지역("전라남도"), 성별("거세"), 계절("여름"). **숫자로 바꿔야(인코딩) 합니다.**

범주형은 다시 둘로:
- **순서 없는(nominal)**: 지역·성별처럼 "어느 게 더 크다"가 없음.
- **순서 있는(ordinal)**: 등급(1++A > 1+A > …)처럼 순서가 있음.

이 구분이 **어떤 인코딩을 쓸지**를 결정합니다.

---

## 3. 인코딩 방법들 — 전체 지도

인코딩(encoding) = **범주(글자)를 숫자로 바꾸는 것.** 방법이 여러 가지인데, 처음엔 다 비슷해 보이지만 **잘못 고르면 모델이 헛것을 배웁니다.** 하나씩, 왜 그 방법이 생겼는지와 함께 봅시다.

### 3-1. 라벨 인코딩(Label Encoding) — 그냥 번호 매기기

가장 단순. 각 범주에 번호를 붙입니다.

```
전라남도 → 0,  경상북도 → 1,  강원도 → 2, ...
```

**문제**: 모델이 이 번호를 **크기·순서로 오해**합니다. "강원도(2)는 전라남도(0)의 2배"라거나 "경상북도(1)가 둘 사이"라고 학습할 수 있죠. 지역엔 그런 순서가 없는데요! 

→ **순서 없는 범주에 라벨 인코딩은 위험**합니다. 단, **트리 모델**(LightGBM 등)은 번호를 크기로 안 쓰고 "쪼개는 기준"으로만 써서 비교적 괜찮습니다. 또 **순서 있는 범주**(등급)엔 오히려 적합합니다 — 순서가 진짜 있으니까. (우리가 등급을 0~15로 바꾼 게 라벨 인코딩이자 순서 인코딩입니다.)

### 3-2. 원-핫 인코딩(One-Hot Encoding) — 칸을 쪼개기

라벨 인코딩의 "가짜 순서" 문제를 푸는 방법. 각 범주를 **0/1짜리 별도 열**로 펼칩니다.

```
지역="전남"  →  [전남=1, 경북=0, 강원=0, ...]
지역="경북"  →  [전남=0, 경북=1, 강원=0, ...]
```

이제 모델은 "전남이냐 아니냐"만 봅니다. 가짜 순서가 없죠. **순서 없는 범주의 표준 방법**이고, 회귀 같은 선형 모델에 특히 적합합니다.

**단점**: 범주 종류가 많으면 **열이 폭발**합니다. 지역(17개)은 17열이라 괜찮지만, 시군구(184개)는 184열, 읍면동(1,306개)은 1,306열이 됩니다. 너무 많으면 비효율적이고 과적합 위험도 커집니다. → 종류 많은 건 다른 방법(빈도/타깃 인코딩)으로.

#### 더미 변수 함정 (drop_first) — 꼭 알아야 할 디테일

원-핫에서 **선형 모델(회귀)** 을 쓸 때 생기는 미묘한 문제입니다.

계절을 [봄,여름,가을,겨울] 4열로 만들면, **3개만 알면 나머지 1개는 자동으로 결정**됩니다(봄=0,여름=0,가을=0이면 무조건 겨울=1). 즉 4번째 열은 **완전히 중복(redundant)** 입니다. 이런 완전 중복이 있으면 회귀가 계수를 못 구합니다(수학적으로 **특이행렬**이 됨 — R3에서 다중공선성으로 다시 만납니다).

> **"특이행렬(singular matrix)"이 뭔가요?** 회귀는 계수를 구할 때 내부적으로 "나눗셈" 비슷한 행렬 연산을 합니다. 그런데 변수 중에 완전 중복(다른 변수로 100% 예측되는 것)이 있으면 그 나눗셈이 **0으로 나누기**처럼 되어버려 답이 안 나옵니다. 이 "0으로 나누기 상태"의 행렬을 특이행렬이라 합니다. 지금은 **"중복 변수가 있으면 회귀가 계수 계산에 실패하는 상태"** 로만 이해하면 됩니다. `drop_first`로 중복 열 하나를 빼면 해결됩니다.

**해결**: 첫 열 하나를 버립니다(`drop_first=True`). 버린 범주(예: 봄)는 **기준점**이 되고, 나머지 계수는 "봄 대비 얼마나 다른가"로 해석됩니다. 정보 손실은 없습니다.

> 트리 모델은 이 함정이 없어서 안 버려도 됩니다. **회귀(선형)에서만** drop_first가 중요합니다.

### 3-3. 빈도 인코딩(Frequency Encoding) — 등장 횟수로

범주를 **"데이터에 몇 번 나오는가"** 로 바꿉니다.

```
KPN_무슨씨수소 → 그 씨수소가 데이터에 9,161번 등장 → 9161
```

종류가 매우 많을 때(혈통처럼) 유용합니다. 열이 1개로 끝나고(원핫처럼 안 터짐), "인기 있는/흔한 범주"라는 정보가 숫자에 담깁니다. 우리 혈통 변수(KPN_NO 등)가 이 방법을 씁니다. **정답(타깃)을 안 쓰므로 누수 위험이 없어** 분할 전에 계산해도 안전합니다.

단점: 빈도가 같은 서로 다른 범주는 구분 못 합니다(둘 다 100번 나오면 똑같은 숫자).

### 3-4. 타깃 인코딩(Target Encoding) — 가장 강력하고 가장 위험

사용자가 예로 든 것입니다. **이게 왜 강력하고 왜 위험한지** 제대로 봅시다.

#### 아이디어

범주를 **"그 범주의 평균 타깃값"** 으로 바꿉니다. 지역을 예로:

```
시군구 A의 소들 평균 등급점수 = 11.2  →  시군구 A를 11.2로
시군구 B의 소들 평균 등급점수 =  8.5  →  시군구 B를 8.5로
```

엄청 똑똑하죠. 184개 시군구를 **단 1개 열**로 압축하면서도, "이 동네 소들은 보통 등급이 좋다/나쁘다"는 **핵심 정보를 직접** 담습니다. 종류 많은 범주에 원핫 대신 쓰는 최고의 방법입니다.

#### 왜 위험한가 — 누수의 교과서적 사례

문제는 **타깃(정답)을 써서 인코딩한다**는 것입니다. [C2](C2_데이터분할과_누수.md)에서 배운 누수가 정확히 여기서 터집니다.

만약 전체 데이터로 시군구 평균 등급을 구하면, 그 평균엔 **검증셋·test 소들의 정답도 섞여** 있습니다. 그 값을 학습에 쓰면 — 모델이 **검증셋의 정답을 미리 엿본** 겁니다(전처리 누수 + 타깃 누수). 특히 그 시군구에 소가 몇 마리 없으면, 평균이 거의 "그 소 자신의 정답"이 되어버려 완전한 컨닝이 됩니다.

#### 올바른 방법: OOF Target Encoding

[C2](C2_데이터분할과_누수.md)의 교차검증 구조를 그대로 씁니다. **각 fold의 값을, 그 fold를 뺀 나머지 데이터로만** 계산합니다.

```
fold 0의 시군구 평균 ← fold 1,2,3,4 데이터로만 계산  (fold 0 자신은 안 봄)
fold 1의 시군구 평균 ← fold 0,2,3,4 데이터로만 계산
...
```

"Out-Of-Fold(자기 fold 밖)"라서 자기 정답을 절대 안 봅니다. test에는 dev 전체로 만든 평균을 적용합니다. 이렇게 하면 누수 없이 타깃 인코딩의 힘을 씁니다.

> 추가 안전장치 **스무딩(smoothing)**: 소가 적은 범주는 평균이 불안정하므로, "그 범주 평균"과 "전체 평균"을 섞어서 극단값을 눌러줍니다. 표본이 적을수록 전체 평균 쪽으로 당겨집니다.

> **우리 프로젝트**: 시군구(184) 타깃 인코딩은 누수 위험 때문에 **18회차(분할 이후)로 미뤘습니다.** 17회차까지는 빈도 인코딩·원핫만 했습니다. 분류 성능을 더 올리고 싶으면 C7에서 OOF 타깃 인코딩을 추가하는 게 1순위입니다.

### 인코딩 방법 비교 (핵심 표)

| 방법 | 무엇으로 바꾸나 | 좋은 경우 | 주의 |
| --- | --- | --- | --- |
| 라벨 | 번호(0,1,2…) | 순서 있는 범주, 트리 모델 | 선형 모델 + 순서 없는 범주엔 가짜 순서 위험 |
| 원-핫 | 0/1 열 여러 개 | 종류 적은 범주(지역17), 선형 모델 | 종류 많으면 열 폭발. 회귀는 drop_first |
| 빈도 | 등장 횟수 | 종류 많은 범주(혈통) | 빈도 같으면 구분 불가. **누수 없음** |
| **타깃** | 범주의 평균 정답 | 종류 많고 정보 중요(시군구) | **누수 위험! 반드시 OOF** |

## 4. 결측치(빈 칸) 처리

우리 데이터엔 빈 칸이 많습니다([병합·전처리 보고서] 참고): 가격 37%, 혈통 38%, 농장 면적 35%… 모델은 빈 칸을 못 먹으니 처리해야 합니다.

### 결측이 생기는 이유부터 (이게 처리 방법을 정함)

- **기록이 아예 없음**: 폐사 기록 없는 농장의 death_count → "0건"이 아니라 "모름".
- **비공개**: 가격(COST_AMT) -99 → 거래가 비공개.
- **해당 없음**: 등외 등급이라 육질 판정을 안 함.

"왜 비었나"에 따라 다르게 다뤄야 합니다. **함부로 0으로 채우면** "모름"과 "진짜 0"을 뒤섞어 버립니다.

### 처리 방법들

| 방법 | 설명 | 언제 |
| --- | --- | --- |
| **그대로 두기(NaN 유지)** | 빈 칸을 빈 칸으로 | **트리 모델**(LightGBM/XGB/CatBoost) — NaN을 알아서 처리함 |
| 중앙값/평균 대치 | 그 열의 중앙값으로 채움 | 선형 모델(회귀·로지스틱) — NaN 못 먹음 |
| 결측 표시 추가 | "이 값은 원래 비었음" 0/1 열을 따로 만듦 | "비었다는 사실 자체가 정보"일 때 |
| 예측 대치 | 다른 변수로 빈 값을 예측해 채움 | 고급, 누수 주의 |

### 핵심 두 가지

1. **트리 3종은 NaN을 그대로 먹습니다.** 채울 필요 없습니다. 오히려 섣불리 채우면 분포를 왜곡합니다.
2. **선형 모델은 NaN을 못 먹습니다.** 채워야 합니다. 이때 **중앙값을 반드시 학습셋에서만** 계산합니다([C2](C2_데이터분할과_누수.md) 전처리 누수!). 우리 코드의 `prepare_X`가 이걸 보장합니다.

> 중앙값 vs 평균: **중앙값을 권합니다.** 평균은 극단값(이상치)에 끌려가지만 중앙값은 안정적입니다(R3에서 "강건성"으로 다시).

## 5. 스케일링(크기 맞추기)

마지막 작업. 변수마다 **숫자의 크기 단위가 천차만별**입니다.

```
월령:       28 ~ 36          (수십 단위)
몸무게:     300 ~ 600        (수백 단위)
더위 비율:  0.0 ~ 1.0        (소수점)
사육밀도:   0.001 ~ 0.01     (아주 작음)
```

### 왜 문제인가 — 선형 모델·거리 기반 모델

선형 모델이나 거리를 재는 모델은 **숫자가 큰 변수에 과도하게 휘둘립니다.** 몸무게(수백)가 더위 비율(소수점)보다 그냥 "숫자가 커서" 더 중요하게 취급될 수 있죠. 단위 때문에 생기는 가짜 중요도입니다.

### 표준화(Standardization) — 가장 흔한 방법

각 변수를 **평균 0, 표준편차 1** 로 바꿉니다.

```
z = (x - 평균) / 표준편차
```

이러면 모든 변수가 "평균에서 몇 표준편차 떨어졌나"라는 **같은 잣대**가 됩니다. 단위가 사라져서 공정하게 비교됩니다. (정규화/Min-Max 스케일링이라는 0~1로 만드는 방법도 있지만, 우리는 표준화를 씁니다.)

### 누가 스케일링이 필요한가? (중요)

| 모델 | 스케일링 |
| --- | --- |
| **선형 모델**(로지스틱·회귀·Lasso/Ridge) | **필요** — 안 하면 단위에 휘둘리고 규제가 불공정 |
| **트리 모델**(LightGBM/XGB/CatBoost) | **불필요** — "크다/작다"의 순서만 보지 크기 자체는 안 씀 |

> 트리는 "몸무게가 450보다 큰가?"로 쪼개기만 하므로 단위를 바꿔도 결과가 똑같습니다. 그래서 **우리 분류(트리 위주)는 스케일링을 로지스틱에만** 적용하고, **회귀(규제 모델)는 표준화가 필수**입니다.

스케일링도 **학습셋에서 평균·표준편차를 구해** 검증·test에 적용합니다(또 전처리 누수 방지!).

---

## 6. 우리 프로젝트 정리

- **인코딩**: 지역·계절·성별 → 원핫(drop_first, 회귀용). 혈통 → 빈도 인코딩. 시군구 → 타깃 인코딩(18회차, OOF). 등급 → 라벨/순서(0~15).
- **결측**: 트리 모델엔 NaN 유지. 선형 모델엔 학습셋 중앙값 대치(`prepare_X`).
- **스케일링**: 트리는 안 함, 로지스틱·규제회귀는 표준화(학습셋 기준).
- 모든 "데이터로 계산하는 값"(중앙값·평균·표준편차·타깃평균)은 **학습셋에서만** → 검증·test엔 적용만. 이게 전처리 누수를 막는 황금률.

---

## 7. 핵심 정리

- 모델은 숫자만 먹는다 → **인코딩(글자→숫자)·결측처리·스케일링**이 필요.
- 변수는 **수치형**(그대로)과 **범주형**(인코딩), 범주형은 **순서 있음/없음**으로 나뉜다.
- 인코딩 4종: **라벨**(번호, 순서형/트리), **원핫**(0/1열, 종류 적을 때, 회귀는 drop_first), **빈도**(등장 횟수, 누수 없음), **타깃**(범주 평균 정답, 강력하지만 **반드시 OOF**).
- 결측: **트리는 NaN 유지**, 선형은 **학습셋 중앙값** 대치. "왜 비었나"를 보고 정한다.
- 스케일링(표준화): **선형 모델만 필요**, 트리는 불필요. 역시 학습셋 기준.
