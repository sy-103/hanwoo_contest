> **이 강의의 목표**: 학습·검증을 마친 모델로 **진짜 대회 test를 예측해 제출 파일**을 만드는 전 과정을 이해하고, 일주일 동안 **점수를 올리는 전략**을 우선순위별로 배웁니다. 그리고 분류 트랙 전체를 총정리합니다.

## 1. test 예측이 왜 까다로운가

[00-B](00B_우리프로젝트_전체그림.md)에서 봤듯이 진짜 test(`test_hanwoo.csv`)는 **정답이 없는** 45만 마리입니다. 그런데 더 큰 문제가 있습니다 — **test엔 원본 13개 컬럼만** 있고, 우리가 만든 92개 피처(기상 일수·daily_gain·더미 등)가 **하나도 없습니다.**

```
train(step11):  92개 피처 (기상·파생·더미 다 있음)  → 모델이 이걸로 학습
test:           13개 원본 컬럼만                    → 피처가 없음!
```

모델은 92개 피처를 받도록 학습됐는데 test엔 13개뿐이니, **test에도 train과 똑같은 병합·전처리·파생을 다시 적용**해야 합니다. 이게 test 예측의 핵심 작업입니다. 한 군데라도 train과 다르게 만들면 모델이 엉뚱한 값을 받아 예측이 무너집니다.

## 2. test 피처를 만드는 3대 누수 방지 원칙

test에 피처를 만들 때 [C2](C2_데이터분할과_누수.md)·[C3](C3_인코딩과_결측_스케일링.md)에서 배운 누수가 또 등장합니다. **세 가지 원칙**을 지킵니다.

### 원칙 ① 통계는 train에서 배워 test에 "적용만"

빈도 인코딩의 빈도 맵, 결측 대치 중앙값, heat_high 임계값 등 **"데이터로 계산하는 값"은 전부 train(dev)에서 구한 것**을 test에 적용합니다. test로 다시 계산하면 안 됩니다.

```python
# 혈통 빈도: train에서 만든 맵을 test에 매핑 (test로 다시 세지 않음)
freq_map = train에서_계산한_빈도
test["KPN_NO_freq"] = test["KPN_NO"].map(freq_map)   # train에 없는 신규 혈통 → NaN (자연스러움)
```

### 원칙 ② 더미 컬럼은 train 목록에 맞춰 정렬

원-핫([C3](C3_인코딩과_결측_스케일링.md))을 test에 그냥 하면, test에만 있는/없는 범주 때문에 **열 개수·순서가 train과 어긋날** 수 있습니다. 모델은 학습 때와 **똑같은 열 구성**을 요구하므로 맞춰야 합니다.

```python
for col in train_더미컬럼들:
    test[col] = test더미[col] if 있으면 else 0   # train에 있는데 test에 없으면 0으로 채움
```

### 원칙 ③ 기상 집계는 같은 날씨 테이블·같은 방식

train과 똑같은 THI 날씨 테이블로, 똑같은 누적합 방식으로 집계합니다(test 관측소 199곳이 전부 날씨 테이블에 있음을 확인). 다른 데이터·다른 방식이면 분포가 어긋납니다.

> 이 세 원칙이 지켜진 `build_features_for_test` 함수가 실습 강의안 [3.5장](../19_분류모델_최종.md)에 검증된 채로 있습니다. **assert로 58개 피처가 다 만들어졌는지 자동 점검**까지 합니다.

## 3. 제출 모델은 데이터를 100% 쓴다

미묘하지만 중요한 디테일. 홀드아웃 평가가 끝났으면, **제출용 최종 모델은 dev+holdout 전체(라벨 100%)로 다시 학습**합니다.

**왜?** holdout(20%)은 "처음 보는 데이터 점수"를 재려고 잠시 떼어둔 것입니다. 그 평가가 끝났으면 그 20%도 **귀한 학습 데이터**입니다. 안 쓰면 데이터의 1/5를 버리는 셈이죠. 그래서:

```
① 튜닝·검증:        dev로 (holdout 안 봄)
② 홀드아웃 점검:    dev로 학습 → holdout 1회 채점 (예상 점수 확인)
③ 제출 모델 학습:   dev + holdout 전체로 다시 학습  ← 라벨 100% 활용
④ test 예측·제출:  ③ 모델로
```

> 비유: 모의고사(holdout)로 실력을 확인했으면, 수능(test) 전엔 모의고사 범위까지 다 복습하고 들어가는 게 당연합니다.

## 4. 제출 파일 점검 체크리스트

예측을 만들었으면 제출 전 반드시 확인:

- [ ] **행 수**가 test와 같은가 (452,497)
- [ ] 예측에 **빈칸(결측)이 없는가**
- [ ] **등급 분포가 train과 비슷한가** — 1++만 잔뜩 나오면 클래스 가중치가 과한 것, 한 등급도 안 나오면 뭔가 이상
- [ ] 대회가 요구하는 **컬럼명·순서·인코딩**에 맞췄는가 (주최측 양식 확인 필수)

## 5. 일주일 동안 점수 올리기 — 우선순위 전략

효과 큰 순서로. 처음 며칠은 1~2번, 후반엔 3~5번에 도전하세요.

### 1순위 — 파라미터 튜닝 확장 ([C5](C5_튜닝과_모델해석.md))

가장 확실하고 쉬운 개선. Optuna 시도 횟수를 늘리고(밤새 50~100회), 탐색 범위를 넓힙니다.

| 손잡이 | 어떻게 |
| --- | --- |
| learning_rate ↓ + n_estimators ↑ | 0.02~0.03 + 2000~3000 (조기 종료 믿고) |
| num_leaves | 31~511로 범위 확장 (데이터 크니 여유) |
| min_child_samples | 20~300 탐색 |

### 2순위 — 클래스 가중치 미세조정 ([C2](C2_데이터분할과_누수.md))

`"balanced"`는 출발점일 뿐. Macro-F1을 직접 노리려면 소수 클래스 가중을 **balanced보다 더 키우거나 줄이며** 최적점을 찾습니다(지수 γ를 0.5~1.5로 탐색).

### 3순위 — 피처 추가: 시군구 OOF 타깃 인코딩 ([C3](C3_인코딩과_결측_스케일링.md))

18회차에 미뤘던 것. 시군구(184종)를 **fold 안에서만** 평균 등급으로 인코딩하면 누수 없이 강력한 피처가 됩니다. 관측소·읍면동도 같은 방식으로 시도. **누수 주의**: 평균은 반드시 해당 fold의 train에서만 계산.

### 4순위 — 모델 앙상블 ([C4](C4_분류모델_4종.md))

LightGBM·XGBoost·CatBoost의 **예측 확률을 평균**해서 최종 결정. 세 모델이 서로 다른 실수를 하므로, 합치면 1~3%p 오르는 경우가 흔합니다. ("전문가 셋의 의견을 종합")

### 5순위 — 문제 구조 바꾸기 (고급)

- **2단계 분해**: 등급을 "숫자등급(1++/1+/1/2/3/등외) × 육량(A/B/C)" 두 모델로 나눠 예측 후 조합. 클래스당 표본이 늘어 소수 등급에 유리할 수 있음.
- **등외 먼저 거르기**: 등외(노폐우 다수)를 이진 분류로 먼저 걸러내고 나머지 15등급 분류.

### 하루 단위 추천 플랜

| 일차 | 할 일 |
| --- | --- |
| 1 | FAST 모드로 끝까지 완주 — 베이스라인 점수 확보 |
| 2 | 전체 5-fold로 4모델 공정 비교 → 메인 확정 |
| 3~4 | Optuna 50+회 + 클래스 가중치 γ 탐색 |
| 5 | 시군구 OOF 타깃 인코딩 추가 |
| 6 | 앙상블·2단계 분해 실험 |
| 7 | 최고 조합 5-fold 재확인 → **홀드아웃 1회** → 제출 |

> **철칙 다시**: 위 실험은 전부 **교차검증 안에서** 비교합니다. holdout은 맨 마지막 1회만. 분할 파일(`split_assignment.csv`)은 절대 새로 만들지 않습니다.

## 6. 분류 트랙 총정리 (C1~C7 한눈에)

데이터 분석 한 사이클을 우리는 이렇게 돌았습니다.

```
[00] 머신러닝·우리 문제 이해 (분류, 트랙1, 데이터3종, 누수)
  ↓
[C1] "잘했다"를 정의   → Macro-F1 (불균형이라 정확도 못 믿음)
  ↓
[C2] 데이터 나누기     → 농장 단위 5-fold + holdout, 누수 4종 차단, 클래스 가중치
  ↓
[C3] 모델 먹이 만들기  → 인코딩(원핫/빈도/타깃)·결측·스케일링
  ↓
[C4] 모델 고르기       → 로지스틱(기준점) + 부스팅 3종(LGB 메인)
  ↓
[C5] 더 좋게·왜 그런지 → Optuna 튜닝 + SHAP 해석
  ↓
[C6] 코드로 실행
  ↓
[C7] test 예측·제출 + 일주일 향상
```

**관통하는 한 가지 원칙**: 모든 단계에서 **"이게 누수는 아닌가? 이게 일반화에 도움 되나?"** 를 물었습니다. 피처 선택도, 분할도, 전처리도, 튜닝도, test 처리도 — 전부 [00-A](00A_데이터분석이란.md)의 **일반화**와 [C2](C2_데이터분할과_누수.md)의 **누수 방지**로 귀결됩니다. 이 두 개를 몸에 익히면 어떤 데이터 분석에도 통합니다.

## 7. 핵심 정리

- test는 정답 없고 원본 13컬럼뿐 → **train과 똑같은 병합·전처리·파생**을 다시 적용.
- test 처리 3원칙: **통계는 train에서 배워 적용만**, **더미는 train 열에 맞춰 정렬**, **기상은 같은 테이블·방식**.
- 제출 모델은 **dev+holdout 전체(100%)로 재학습** — 홀드아웃 평가 후엔 그 데이터도 학습에 쓴다.
- 제출 전 **행수·결측·분포·양식** 점검.
- 향상 우선순위: **튜닝 → 클래스 가중 → OOF 타깃 인코딩 → 앙상블 → 구조 변경.** 전부 CV 안에서, holdout은 1회.
- 분류 트랙 전체를 관통하는 원칙: **일반화 + 누수 방지.**
