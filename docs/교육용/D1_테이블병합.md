# D1. 데이터 준비 ① — 흩어진 테이블 합치기 (병합)

> **이 강의의 목표**: 모델에 쓸 데이터는 처음부터 한 덩어리가 아닙니다. 소·농장·혈통·날씨가 **여러 파일에 흩어져** 있죠. 이걸 하나로 합치는 **병합(merge/join)** 의 원리를 배웁니다. 키(key)란 무엇인지, 왜 "단위"를 맞춰야 하는지, 합칠 때 **행이 늘거나 줄면 안 되는** 이유, 그리고 우리가 실제로 5개 원본을 240만 × 44 테이블로 합친 과정을 봅니다.
> **앞 강의**: [00-B](00B_우리프로젝트_전체그림.md)의 데이터 구성.

---

> 🗺️ **학습 여정**: 기초(00A·00B) → 데이터준비(D1·D2) → EDA(E1·E2·E3) → 〔분류 C1–C7〕 · 〔회귀 R1–R8〕  ·  **📍 지금: 데이터준비 1/2**

## 1. 왜 합치나 — 데이터는 흩어져 있다

현실의 데이터는 보통 **여러 파일(테이블)에 나뉘어** 있습니다. 한 곳에 다 넣으면 중복이 많아 비효율적이라, **주제별로 따로** 관리하기 때문입니다. 우리 한우 데이터도 그렇습니다.

| 원본 파일 | 한 줄이 뜻하는 것 | 담긴 정보 |
| --- | --- | --- |
| `hanwoo_train` | 소 한 마리 | 등급·체격·지역·날짜 (기준 테이블) |
| `hanwoo_lineage` | 소 한 마리 | 혈통(부·모·조부모) |
| `hanwoo_death` | 폐사 1건 | 농장별 폐사 기록 |
| `hanwoo_area` | 농장 허가 1건 | 농장 면적·사육두수 |
| `hanwoo_weather` (+ASOS) | 관측소의 하루 | 일별 기온·습도·강수 |

모델은 **한 마리당 모든 정보가 한 줄에** 있어야 학습할 수 있습니다. 그래서 흩어진 이 정보들을 **소 기준 한 테이블로** 모읍니다. 이게 병합입니다.

> 비유: 한 학생의 생활기록부를 만드는데 성적표·출결부·건강기록이 각각 다른 서류철에 있다면? **학번을 기준으로** 세 서류철에서 그 학생 정보를 찾아 한 장에 옮겨 적어야 합니다. 그 "학번"이 곧 병합의 **키**입니다.

---

## 2. 키(key) — 무엇을 기준으로 잇나

**키(key)** = 두 테이블에서 "같은 대상"을 알아보는 **공통 열**입니다. 키가 같은 행끼리 이어 붙입니다. 우리 데이터의 키는 셋입니다.

```
소 단위 정보(혈통)        → CATTLE_NO (소 고유번호)로 잇는다
농장 단위 정보(폐사·면적)  → FARM_UNIQUE_NO (농장 고유번호)로 잇는다
날씨                     → stn(관측소) + date(날짜)로 잇는다
```

**무엇을 키로 잡느냐가 병합 설계의 핵심**입니다. 혈통은 소마다 다르니 소 번호로, 폐사·면적은 농장 단위 정보라 농장 번호로 잇습니다.

---

## 3. 조인의 종류 — LEFT JOIN을 쓰는 이유

두 테이블을 키로 잇는 걸 **조인(join)** 이라 합니다. 방식이 여러 가지인데, 우리는 **LEFT JOIN(왼쪽 우선 병합)** 을 씁니다.

### LEFT JOIN이란

**기준 테이블(왼쪽 = train)의 행은 하나도 잃지 않고**, 오른쪽 테이블에 짝이 있으면 값을 가져오고 **없으면 빈칸(NaN)** 으로 둡니다.

```
train(소)        lineage(혈통)         결과(LEFT JOIN)
─────────        ──────────           ──────────────
소1 ─ 키:A   +   키:A 혈통=KPN001  →  소1, 혈통=KPN001
소2 ─ 키:B       키:C 혈통=KPN002  →  소2, 혈통=NaN  (B는 혈통표에 없음)
소3 ─ 키:C                          →  소3, 혈통=KPN002
```

### 왜 LEFT JOIN인가

우리 목표는 **"train의 모든 소"** 에 정보를 붙이는 것입니다. 혈통이 없는 소라도 **버리면 안 됩니다**(등급은 있으니 학습에 써야죠). LEFT JOIN은 기준(train) 행을 100% 보존하므로 딱 맞습니다.

> 다른 조인: **INNER JOIN**은 양쪽에 다 있는 행만 남깁니다(혈통 없는 소는 사라짐 → 안 됨). **OUTER JOIN**은 양쪽 모두 살립니다. 우리는 train을 기준으로 보존하는 LEFT가 정답입니다.

---

## 4. 가장 중요한 규칙: 행 수가 변하면 안 된다

병합에서 **초보가 가장 크게 망하는** 지점입니다. 꼭 기억하세요.

> **LEFT JOIN 후 행 수는 기준 테이블과 똑같아야 한다.** 우리는 매 병합 후 **"행 수 = 2,408,699"** 를 확인합니다.

- **행이 늘었다** → 키가 **중복**됐다는 신호(1:N 병합 사고). 예: 한 농장에 면적 허가가 3건이면, 농장 키로 그냥 붙일 때 소 한 마리가 3줄로 복제됩니다.
- **행이 줄었다** → 실수로 INNER JOIN을 했다는 신호(짝 없는 행이 삭제됨).

### 단위를 맞추는 법 — 집계 먼저, 병합 나중

행 증가(1:N)를 막는 핵심 기술입니다. train은 "**소 1마리 = 1행**"인데, death는 "**폐사 1건 = 1행**", area는 "**허가 1건 = 1행**"입니다. 단위가 다르죠. 이걸 그대로 붙이면 행이 불어납니다.

→ 그래서 **먼저 농장 단위로 집계(groupby)해서 1:1로 만든 뒤** 붙입니다.

```
death(폐사 건별)                농장별 집계               train에 붙이기
──────────────                ──────────              ────────────
농장X 폐사기록             →    농장X: death_count=5  →  소(농장X) ← 5
농장X 폐사기록                  (count로 1행으로 압축)    소(농장X) ← 5
농장X 폐사기록
```

폐사는 **건수(count)**, 면적·사육두수는 **합(sum)** 으로 집계해 농장당 1행으로 만든 뒤 LEFT JOIN합니다. 이러면 행이 안 늘죠.

> 한 줄: **단위가 다르면(1:N) 집계로 1:1을 만든 뒤 붙인다. 그리고 항상 행 수를 확인한다.**

---

## 5. 날씨 합치기 — 가장 까다로운 부분 (생애 누적)

날씨는 단순 키 매칭이 아닙니다. **"이 소가 살아온 기간(출생~도축) 동안 더위에 며칠 노출됐나"** 를 세야 하니까요. 소마다 기간이 다르고, 그 기간의 날씨를 다 더해야 합니다.

문제: 소 240만 마리 × 평균 1,000일 = 수십억 번 날짜를 일일이 세면 며칠이 걸립니다. 해결책이 **누적합(prefix sum)** 이라는 기법입니다.

```
관측소별로 "1월 1일부터 누적 더위일수"를 미리 만들어 두면,
어떤 소의 (출생~도축) 더위일수 = 누적[도축일] − 누적[출생일]   ← 뺄셈 한 번!
```

미리 누적표를 만들어 두고 **뺄셈 한 번**으로 각 소의 생애 노출을 구하니, 240만 마리가 몇 초 만에 끝납니다. (THI가 무엇인지는 [E3](E3_EDA_기상_시공간_파생.md)에서 자세히 — 여기선 "생애 기간만큼의 날씨를 효율적으로 더했다"만 알면 됩니다.)

> 이렇게 만든 `days_양호/주의/경고/위험`(생애 더위 등급별 일수)이 나중에 EDA·모델의 핵심 기상 변수가 됩니다.

---

## 6. 병합 결과 — 우리가 실제로 만든 것

5개 원본(+추가 기상)을 위 원칙대로 합쳐 **한 마리당 한 줄**짜리 테이블을 완성했습니다.

```
hanwoo_train (240만 × 23)
  + death 집계   → death_count 1열
  + lineage      → 혈통 7열
  + area 집계    → 면적·사육두수 4열
  + weather 누적 → 생애 기상 9열
  ─────────────────────────────────
  = 2,408,699행 × 44열  (행 수 그대로! ✓)
```

각 병합마다 행 수가 2,408,699로 유지됐는지 확인했고, 무작위 표본의 값이 원본과 일치하는지도 대조했습니다(검증 = 재계산 후 비교).

### 병합하면 빈칸(결측)이 생긴다 — 자연스러운 일

LEFT JOIN의 특성상, 짝이 없는 소·농장은 빈칸이 됩니다. 이건 실패가 아니라 **정보 그 자체**입니다.

- 혈통표에 없는 소 → 혈통 7열이 NaN (미등록 개체)
- 폐사기록 없는 농장 소속 소 → death_count가 NaN (**"폐사 0"이 아니라 "모름"**)

이 빈칸을 어떻게 다루느냐가 다음 강의 **[D2. 전처리](D2_전처리.md)** 의 핵심 주제입니다. ("기록 없음 ≠ 진짜 0" — [E1](E1_EDA_단변량_분포.md)에서 예고한 그 이야기입니다.)

## 7. 핵심 정리

- 데이터는 **여러 테이블에 흩어져** 있고, 모델용으로 **소 기준 한 테이블로 합친다**(병합).
- **키(key)** = 같은 대상을 잇는 공통 열: 소(CATTLE_NO)·농장(FARM_UNIQUE_NO)·날씨(stn+date).
- **LEFT JOIN** = 기준(train) 행을 100% 보존, 짝 없으면 NaN. 행을 안 버려서 우리에게 맞다.
- **철칙: 병합 후 행 수가 변하면 안 된다.** 늘면 키 중복(1:N), 줄면 INNER 사고.
- 단위가 다르면(1:N) **집계(count/sum)로 1:1을 만든 뒤** 붙인다.
- 날씨는 **누적합**으로 생애 노출을 뺄셈 한 번에 집계.
- 결과: **240만 × 44**. 병합으로 생긴 빈칸은 "정보"이며, 처리는 D2에서.
